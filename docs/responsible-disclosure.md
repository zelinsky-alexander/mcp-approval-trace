# Responsible disclosure

Before publishing a potential client failure:

1. Reproduce it twice using a clean profile.
2. Record exact versions and configuration.
3. Verify the evidence bundle.
4. Check current documentation and known issues.
5. Contact the maintainer privately with the smallest reproducible bundle.
6. Allow a reasonable response window appropriate to severity.
7. Offer to retest a fix or mitigation.
8. Publish the maintainer response accurately.

Use harmless markers only. Do not include credentials, private prompts, destructive tool calls, or claims that a model executed an attack unless separately demonstrated and authorized.

Public wording should distinguish:

- UI transparency limitation;
- consent-conformance concern;
- security weakness;
- confirmed exploitable vulnerability.
