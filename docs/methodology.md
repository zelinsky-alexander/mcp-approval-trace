# Research methodology

## Hypothesis

Some MCP clients may send security-relevant tool metadata to a model that is absent, truncated, transformed, or not renewed in their human approval flow.

## Independent variables

- client and exact version;
- scenario;
- initial versus mutated phase;
- notified versus silent mutation;
- client configuration and profile state.

## Dependent variables

- server-to-model field preservation;
- approval visibility by field class;
- warning and diff presentation;
- reapproval behavior;
- changed outbound requests before reapproval;
- code-point preservation or normalization.

## Controls

AT-001 is the clean negative control. Every client must pass infrastructure validation before security scenarios are interpreted. Scenario markers are deterministic, harmless, and do not depend on model behavior.

## Evidence sources

1. Exact scenario YAML and server tool output.
2. Client approval screenshot and structured observer record.
3. Exact outbound model API request captured locally.
4. Deterministic comparison and Unicode analysis.
5. SHA-256 manifest for the publication bundle.

## Reproducibility

A public result must include the ApprovalTrace commit, exact client version, configuration hash, environment metadata, scenario, redacted request, observation record, and evidence manifest. Potential failures require a second clean-profile reproduction.

## Limitations of inference

The capture endpoint observes the request generated by the client. It does not prove provider receipt, provider-side transformations, tokenization, model compliance, tool invocation, or real-world impact.
